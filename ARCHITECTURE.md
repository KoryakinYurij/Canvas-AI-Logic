# Архитектура: Интерактивный ИИ‑канвас

## 1. Обзор

Этот документ описывает многослойную архитектуру приложения, вдохновленную Clean Architecture и адаптированную под Atomic Design. Цель — обеспечить максимальную модульность, масштабируемость и простоту поддержки.

## 2. Четырехслойная Архитектура

Архитектура состоит из четырех независимых слоев, каждый из которых имеет свою зону ответственности. Данные текут от внешних слоев (Infrastructure, Presentation) к ядру (Domain).

```
+---------------------------------------------------+
|            Уровень 1: Presentation Layer          |
| (Визуализация, Atomic Design: Atoms, Molecules)   |
+---------------------------------------------------+
|               Уровень 2: Widget Layer             |
|  (Управление состоянием UI, Layout Engines)       |
+---------------------------------------------------+
|                Уровень 3: Domain Layer            |
|       (Бизнес-логика, Сущности: Node, Graph)      |
+---------------------------------------------------+
|            Уровень 4: Infrastructure Layer        |
|            (Взаимодействие с внешними API)        |
+---------------------------------------------------+
```

### Уровень 1: Presentation Layer (Слой Представления)

*   **Ответственность:** Только рендеринг UI. Этот слой "глупый" — он не знает о бизнес-логике или источниках данных.
*   **Принцип:** Atomic Design.
*   **Иерархия компонентов:**
    *   **Atoms (Атомы):** Базовые, неразделимые UI-элементы.
        *   `Button`, `Icon`, `Spinner`, `NodeShell` (простой прямоугольник с тенью), `PortSocketUI` (кружок для порта).
        *   Принимают `props`, возвращают `onClick` и другие события.
    *   **Molecules (Молекулы):** Комбинации атомов, выполняющие конкретную задачу.
        *   `NodeHeader`: `Icon` + `Title` + `Toolbar` (содержащий `Button`-ы).
        *   `Port`: `PortSocketUI` + `Label`.
    *   **Organisms (Организмы):** Сложные компоненты, собранные из молекул и атомов.
        *   `VisualNodeCard`: Собирает `NodeHeader`, тело ноды и `Port`-ы в единый визуальный блок. Не содержит логики, только верстку.

### Уровень 2: Widget Layer (Слой Виджетов)

*   **Ответственность:** Связывание UI с данными и действиями пользователя.
*   **Компоненты:**
    *   **Smart Components (Контейнеры):**
        *   `NodeContainer`: Обертка, которая подписывается на state-менеджер, получает данные конкретной ноды (`Node Entity`) и ее координаты, а затем передает их в `VisualNodeCard` для рендеринга.
        *   `CanvasWidget`: Управляет глобальным состоянием канваса — зумом, панорамированием, выделением. Содержит в себе все `NodeContainer`-ы.
    *   **Layout Engines (Движки Расположения):**
        *   `AutoLayoutService`: Сервис, который принимает `GraphModel` и с помощью алгоритмов (Dagre, Elkjs) вычисляет оптимальные `x, y` координаты для каждой ноды. Вызывается после генерации или изменения графа.

### Уровень 3: Domain Layer (Слой Домена)

*   **Ответственность:** Бизнес-логика и правила приложения. Это ядро системы.
*   **Сущности:**
    *   **Node Entities:** Классы или объекты, описывающие типы нод и их правила. Например, `FilterNode` может иметь правило "1 вход, 2 выхода". Здесь же определяется логика валидации соединений.
    *   **Graph Model:** Центральная структура данных (`{ nodes: [], edges: [] }`), которая является единственным источником правды о состоянии графа.

### Уровень 4: Infrastructure Layer (Слой Инфраструктуры)

*   **Ответственность:** Взаимодействие с внешним миром.
*   **Компоненты:**
    *   **AI Connector:** Сервис, инкапсулирующий логику общения с API (OpenAI, Claude). Он отправляет запрос пользователя, получает ответ и преобразует его в `GraphModel`, понятный для Domain Layer. Этот слой полностью изолирован, что позволяет легко менять AI-провайдера в будущем.
